# 언리얼 스터디 — 실습용 게임 아이디어 추천

> 작성일: 2026-04-11  
> 기반: 강의 계획 수정본 v4 (4~8주차 커리큘럼)

---

## 추천 전략

4주차부터 8주차까지 **매주 배운 내용을 하나의 프로젝트에 누적 적용**하는 구조가 수정본의 핵심 설계 원칙이다. 따라서 아래 기준으로 게임 아이디어를 선정했다:

1. **커리큘럼 커버리지**: 4~8주차의 모든 주요 주제(입력, HUD, C++/BP, 델리게이트, 컴포넌트, DataAsset, 코딩 컨벤션)를 자연스럽게 사용하는가
2. **점진적 확장**: 매주 기능을 하나씩 얹었을 때 자연스러운 완성도 증가가 체감되는가
3. **토이 스케일**: 5주(실질 실습 시간 약 5~6시간) 안에 눈에 보이는 결과물이 나오는가
4. **흥미 유발**: 수강생이 "이걸 내가 만들었다"는 성취감을 느낄 수 있는가

---

## 추천안 A — 탑다운 액션 서바이벌 (핵심 추천)

> 레퍼런스: Vampire Survivors, Brotato, HoloCure

### 왜 이 장르인가

- **구조가 단순하되 시스템이 풍부하다**: 캐릭터 이동 + 자동/반자동 공격 + 적 웨이브 스폰 + 아이템 획득이라는 명확한 루프가 있어, 매주 하나의 시스템을 추가하기에 최적이다.
- **아트 부담이 낮다**: 탑다운 시점이라 복잡한 3D 애니메이션 없이 기본 메시 + 마테리얼 색상만으로 프로토타입이 가능하다.
- **모든 커리큘럼 주제를 자연스럽게 소화한다** (아래 매핑 참고).

### 주차별 기능 매핑

| 주차 | 커리큘럼 주제 | 프로젝트에 추가할 기능 |
|---|---|---|
| 4주차 (4/27) | PlayerController, PlayerState, HUD | Enhanced Input으로 탑다운 이동 구현, PlayerState에 HP/경험치 저장, HP 바 HUD 위젯 |
| 5주차 (5/4) | BP + C++ 문법 | C++로 Enemy 기본 클래스 작성(UPROPERTY 노출), BP로 Enemy 파생 클래스 제작 → C++/BP 비교 체험 |
| 6주차 (5/11) | 델리게이트, 컴포넌트, 통신 | HealthComponent 분리, 데미지 → 델리게이트 → HUD 자동 갱신, 트리거 볼륨으로 아이템 획득 |
| 7주차 (5/18) | DataAsset, 에셋 관리 | 무기/아이템 DataAsset 테이블 작성, 레벨업 시 DataAsset 기반 무기 선택 UI |
| 8주차 (5/25) | 코딩 컨벤션 + 통합 | 전체 리팩토링, 적 웨이브 시스템 마무리, 게임오버/재시작 흐름 완성 |

### 최종 결과물 이미지

플레이어가 탑다운 시점에서 움직이며, 주변에서 적이 웨이브로 몰려오고, 적을 처치하면 경험치를 얻고, 레벨업 시 무기/버프를 선택하는 2~3분짜리 서바이벌 루프.

### 난이도: ★★☆☆☆

## 추천안 B — 3인칭 던전 크롤러 (한 방 탐험)

> 레퍼런스: 심플한 Diablo 스타일, 혹은 작은 규모의 Zelda 던전

### 왜 이 장르인가

- **3인칭 카메라 + 캐릭터 이동**이 언리얼의 가장 전형적인 학습 경로이며, 공식 튜토리얼/샘플과도 연계하기 좋다.
- **방(Room) 구조**가 있어 레벨 디자인 개념도 자연스럽게 접할 수 있다.
- 트리거 볼륨(문 열기, 함정 발동)과 델리게이트 활용이 직관적이다.

### 주차별 기능 매핑

| 주차 | 프로젝트에 추가할 기능 |
|---|---|
| 4주차 | 3인칭 캐릭터 이동/점프, HP HUD, 카메라 설정 |
| 5주차 | C++로 무기 클래스 작성, BP로 공격 몽타주 연결 |
| 6주차 | HealthComponent, 트리거 볼륨으로 문 열기/함정, 데미지 델리게이트 |
| 7주차 | DataAsset으로 무기/아이템 스탯 정의, 보물상자에서 랜덤 아이템 드롭 |
| 8주차 | 던전 2~3개 방 연결, 보스 방 + 게임 클리어 흐름, 리팩토링 |

### 난이도: ★★★☆☆

3인칭 카메라 세팅과 애니메이션 연동에 약간의 추가 시간이 필요하지만, 언리얼의 ThirdPerson 템플릿을 기반으로 시작하면 부담이 크게 줄어든다.

## 추천안 C — 타워 디펜스 (시스템 중심)

> 레퍼런스: Bloons TD, Kingdom Rush (단순화 버전)

### 왜 이 장르인가

- **컴포넌트 기반 설계의 교과서적 사례**: 타워를 컴포넌트 조합(공격, 범위 감지, 업그레이드)으로 만들면 6주차 컴포넌트 설계를 가장 깊이 체험할 수 있다.
- **DataAsset 활용도가 극대화**: 타워 종류, 적 종류, 웨이브 구성 등 데이터 테이블이 핵심이므로 7주차 내용이 프로젝트의 근간이 된다.
- 플레이어 캐릭터 조작이 없거나 최소화되어 **시스템 설계에 집중** 가능.

### 주차별 기능 매핑

| 주차 | 프로젝트에 추가할 기능 |
|---|---|
| 4주차 | 탑다운 카메라, 클릭으로 타워 배치 UI(커서 + HUD 자원 표시), PlayerState에 골드 저장 |
| 5주차 | C++로 타워 기본 클래스, BP로 타워 파생(근거리/원거리), UPROPERTY로 에디터에서 스탯 조정 |
| 6주차 | 공격 컴포넌트·범위 감지 컴포넌트 분리, 적 사망 → 델리게이트 → 골드 획득, 경로 이동 AI |
| 7주차 | DataAsset으로 타워/적/웨이브 정의, 데이터 드리븐 밸런싱 |
| 8주차 | 웨이브 시스템 완성, 승/패 조건, 코드 정리 |

### 난이도: ★★☆☆☆

캐릭터 이동이 없어 입력 처리가 단순한 대신, 시스템 설계 비중이 높아 프로그래밍 학습에 특화된 선택지다.

## 추천안 D — 복수 토이 프로젝트 (매주 새 미니게임)

> 매주 독립된 토이 프로젝트를 만들어 해당 주차의 주제에 가장 적합한 장르를 선택하는 방식

### 주차별 미니게임 예시

| 주차 | 미니게임 | 학습 포인트 |
|---|---|---|
| 4주차 | **탑다운 슈터** (트윈스틱) | 입력 시스템, HP HUD |
| 5주차 | **클리커 게임** | C++ ↔ BP 연동, UPROPERTY/UFUNCTION |
| 6주차 | **함정 미로** | 트리거 볼륨, 델리게이트 체인 |
| 7주차 | **카드 뽑기 시뮬레이터** | DataAsset, 데이터 드리븐 설계 |
| 8주차 | **가장 재미있었던 프로젝트 확장** | 통합, 리팩토링 |

### 장단점

- **장점**: 매주 새로운 흥미, 해당 주제에 100% 최적화된 실습, 실패해도 다음 주에 리셋
- **단점**: 기능 누적의 성취감이 없고, 매주 프로젝트 세팅에 시간 소모. 수정본의 "누적 프로젝트" 설계 원칙과 상충

## 최종 권장

### 1순위: 추천안 A — 탑다운 액션 서바이벌

- 커리큘럼의 모든 주제를 빠짐없이 커버하면서도, 매주 눈에 보이는 기능이 추가되어 성취감이 극대화된다.
- Vampire Survivors 같은 게임이 대학생 사이에서 인지도가 높아 "이런 걸 만들고 있다"는 동기 부여가 강력하다.
- 탑다운 시점은 카메라/애니메이션 복잡도가 낮아 **시스템 구현에 집중**할 수 있다.
- 기본 메시(큐브, 스피어)와 색상만으로 프로토타입이 가능해 아트 병목이 없다.

### 2순위: 추천안 C — 타워 디펜스 (프로그래밍 학습 특화)

- 컴포넌트 설계와 DataAsset 활용을 가장 깊이 경험할 수 있어, 프로그래밍 역량 강화에 초점을 둔다면 이 선택이 맞다.
- 다만 "게임을 플레이하는 재미"는 A안보다 느리게 체감될 수 있다.

### 하이브리드 옵션

시간이 허용된다면 **A안을 메인 프로젝트로 진행하면서, 4주차에만 별도 토이(추천안 D의 탑다운 슈터)**를 만들어 감을 잡은 뒤, 5주차부터 본격적으로 서바이벌 프로젝트에 기능을 누적하는 방식도 효과적이다.
