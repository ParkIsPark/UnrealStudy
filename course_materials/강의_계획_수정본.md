# 2026-1 언리얼 스터디 강의 계획 수정본

> 원본 강의 계획서 + 1~3주차 실제 진행 내용을 종합하여 작성한 수정본입니다.

---

## 1. 진행 현황 분석 (1~3주차)

### 1주차 (3/23) — 언리얼 엔진 소개
- **계획**: 언리얼 엔진의 기초, 기본 문법, 상속
- **실제 진행**: 엔진 소개 중심. 왜 언리얼을 쓰는가(C++ Native, 블루프린트, 오픈소스, 아티스트 친화적 생태계), 유니티 대비 언리얼의 설계 철학(엄격한 프레임워크), 모던 객체지향(합성 > 상속, 인터페이스 기반, 리플렉션) 개념 소개
- **차이점**: C++ 기본 문법/상속 실습은 진행하지 않음. 엔진 철학과 동기부여에 집중

### 2주차 (3/30) — GameMode / GameState
- **계획**: 월드 생성 과정, GameMode/GameState, 라이프사이클
- **실제 진행**: 계획대로 진행. GameInstance → UWorld → GameMode 스폰 과정, GameMode의 역할(Modern GameManager), 멀티플레이어에서의 서버/클라이언트 구조, Replication 개념, GameState의 복제와 상태 공유 원리까지 심도있게 다룸
- **차이점**: 멀티플레이 관점의 설명이 계획보다 깊었음. 라이프사이클 전체를 다루기보다 GameMode/GameState에 집중

### 3주차 (4/6) — Actor 시스템과 캐릭터
- **계획**: AActor/UObject, Pawn/Character, PlayerCharacter/PlayerState (실습)
- **실제 진행**: UObject → AActor → APawn → ACharacter 상속 계층 전체 설명. UObject 생성 과정(NewObject, 엔진 등록, GC), AActor의 월드 종속성과 Transform, Pawn의 빙의(Possess) 개념, ACharacter와 CharacterMovementComponent 소개. Pawn vs Character 비교 실습 진행
- **차이점**: PlayerState는 다루지 않음. PlayerController 개념은 간략히 언급만 함

### 종합 평가
- 이론적 기반이 탄탄하게 진행됨 (특히 "왜 이렇게 설계했는가"에 집중)
- 3주차 말미에 강사가 직접 언급한 **아직 다루지 않은 것**: UI 제작, 객체 간 통신, 유지보수 가능한 설계, 게임 규모 확장
- 실습 비중을 늘리고 싶다는 강사의 의지가 있음
- PlayerState, PlayerController 상세 내용은 아직 미진행

## 3. 주차별 상세 내용

### 4주차 (4/27) — PlayerController & Enhanced Input 심화 + 실습

**변경 이유**: 3주차에서 Pawn/Character를 다뤘지만 PlayerController와 입력 시스템 상세는 미진행. 원본 계획의 HUD/블루프린트보다 입력 처리를 먼저 다루는 것이 자연스러운 흐름.

**내용**:
- PlayerController의 역할과 Possess 메커니즘 상세
- PlayerState: 플레이어별 상태 관리 (3주차 미진행분 보충)
- Enhanced Input 시스템 심화: InputAction, InputMappingContext, Modifiers/Triggers
- **실습**: 커스텀 입력 바인딩으로 캐릭터 조작 확장 (달리기, 상호작용 키 등)

### 5주차 (5/4) — HUD, 위젯 블루프린트, 블루프린트 기초 + 실습

**변경 이유**: 원본 4/27 내용을 한 주 뒤로 이동. 입력 → UI 순서가 더 논리적.

**내용**:
- UMG(Unreal Motion Graphics) 기초
- 위젯 블루프린트로 HUD 만들기
- 블루프린트 노드 기초 (변수, 이벤트, 함수)
- C++와 블루프린트의 연동 (BlueprintCallable, BlueprintImplementableEvent)
- **실습**: 체력바/상태 표시 HUD 제작

### 6주차 (5/11) — 언리얼 C++ 심화: GC, 리플렉션, 코딩 컨벤션

**내용** (원본 유지):
- 가비지 콜렉터 동작 원리 (2주차에서 개념만 소개한 부분의 심화)
- 리플렉션 시스템: UPROPERTY, UFUNCTION, UCLASS 매크로의 의미
- 언리얼 코딩 컨벤션 (네이밍, 매크로 사용 규칙)
- **실습**: 기존 코드를 컨벤션에 맞게 리팩토링

### 7주차 (5/18) — 설계 방법론: 델리게이트, 컴포넌트 기반 설계 + 실습

**변경 이유**: 3주차 말미에 "객체 간 통신"과 "유지보수 가능한 설계"가 미다룬 주제로 언급됨. 이 주차에서 해결.

**내용**:
- 델리게이트 (싱글/멀티캐스트, 다이나믹): 객체 간 느슨한 결합
- 컴포넌트 기반 설계: ActorComponent, SceneComponent
- 1주차에서 소개한 "합성 > 상속" 원칙의 실제 적용
- **실습**: 상호작용 시스템 구현 (델리게이트로 이벤트 전파)

### 8주차 (5/25) — 에셋 관리 + 데이터 중심 설계 (통합)

**변경 이유**: 원본에서 2주에 걸쳐 다루던 에셋 관리를 1주로 통합. TObjectPtr/SoftPtr 등 포인터 타입은 핵심만 다루고, DataAsset과 데이터 중심 설계에 더 비중을 둠. PrimaryAsset은 간략 소개로 유지.

**내용**:
- 에셋 참조 방식: TObjectPtr, TSoftObjectPtr, TSubclassOf
- DataAsset을 활용한 데이터 중심 설계
- DataTable 활용 (스프레드시트 기반 데이터 관리)
- PrimaryAsset 개념 소개 (간략)
- **실습**: DataAsset으로 캐릭터 스탯/아이템 시스템 구성

### 9주차 (6/1) — 종합 실습 & 마무리

**변경 이유**: 원본에서는 기말고사 휴강이었으나, 에셋 관리 2주분을 통합하여 확보한 1주를 종합 실습에 할당. 강사가 강조한 "직접 만들어보는 경험"과 3주차에서 언급한 "게임 규모 확장" 주제를 다룸.

**내용**:
- 1~8주차 내용을 활용한 미니 프로젝트 실습
- 게임 규모 확장 시 고려사항 (레벨 스트리밍, 모듈 구조화)
- 이후 학습 로드맵 안내 (애니메이션, AI, 네트워크 심화 등)
- Q&A 및 피드백

## 5. 교수법 제안

### 실습 비중 확대
3주차 자료에서 강사가 "실습을 조금 늘려보긴 할 것"이라고 언급한 점을 반영하여, 4주차부터 매 수업 후반 30~40분을 실습 시간으로 배정할 것을 권장합니다.

### 이론-실습 연결 구조
각 주차의 이론이 실습 프로젝트에 누적 반영되는 구조를 권장합니다:
- 4주차: 캐릭터 입력 확장 → 5주차: 여기에 HUD 추가 → 7주차: 상호작용 시스템 추가 → 8주차: 데이터 에셋으로 구조화

### 미니 프로젝트 연계
종합 실습(9주차)에서 사용할 미니 프로젝트 주제를 4주차에 미리 공개하면, 학생들이 매주 배운 내용을 하나의 프로젝트에 점진적으로 적용하는 경험을 할 수 있습니다.
