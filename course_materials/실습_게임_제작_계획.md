# 언리얼 스터디 — 실습용 게임 제작 계획서

> 작성일: 2026-04-11  
> 기반: 강의 계획 수정본 v4 + 실습 게임 아이디어 추천 v2  
> 프로젝트명: **ModularSurvivor** (가칭)

---

## 1. 프로젝트 개요

### 한 줄 정의

> "언리얼 프레임워크의 설계 의도 — GameMode, 컴포넌트, DataAsset — 를 체감하기 위한 뱀서류 기반 탑다운 서바이벌 게임"

### 제작 목적

이 프로젝트는 **완성도 높은 게임을 만드는 것이 아니라**, 아래 학습 목표를 체감하는 데 목적이 있다:

| 학습 목표 | 체감 방식 |
|---|---|
| GameMode / GameState 분리 이유 | 같은 맵·캐릭터에서 GameMode 3개로 완전히 다른 게임을 경험 |
| 컴포넌트 기반 설계 | 컴포넌트 교체만으로 캐릭터/적의 행동이 달라지는 것을 직접 확인 |
| DataAsset 데이터 드리븐 | 코드 한 줄 안 고치고 에디터에서 밸런스 조정 |
| C++ + 블루프린트 협업 | 같은 기능을 C++로도, BP로도 구현해 비교 체험 |
| 델리게이트 통신 | 오브젝트 간 직접 참조 없이 이벤트로 소통하는 구조 체험 |

### 아트 방침

- **프로토타입 메시만 사용** (별도 아트 에셋 불필요)
  - 플레이어: 기본 UE5 Mannequin 또는 파란 캡슐
  - 적: 색상으로 구분 (빨간 스피어 = 추적형, 주황 큐브 = 돌진형, 보라 콘 = 원거리형)
  - 무기 이펙트: DrawDebugSphere / DrawDebugLine 등 디버그 도형
  - 맵: 기본 평면 + 박스 메시로 경계벽
- 아트 없이 **시스템 설계에 집중**하는 것이 핵심

## 3. 아키텍처 설계

### 3.1 전체 클래스 구조

```
AGameModeBase (x3)
 ├── AMS_GM_Survival      — 무한 웨이브, 생존 시간 기록
 ├── AMS_GM_TimeAttack    — 제한 시간, 킬 수 경쟁
 └── AMS_GM_BossRush      — 보스만 등장, 클리어 수 기록

AGameStateBase (x3 또는 공통 1개)
 └── AMS_GameState        — 시간, 킬 수, 웨이브, 점수 등 공용 상태

ACharacterBase (ACharacter 상속)
 └── AMS_PlayerCharacter
      ├── UWeaponComponent       (교체 가능: 근접/원거리/범위)
      ├── UHealthComponent       (플레이어·적 공용)
      ├── UPassiveBuffComponent  (런타임 부착/제거)
      └── UMovementModifierComponent

AMS_EnemyBase (AActor 상속)
 ├── AMS_Enemy_Chaser     — 추적형 (파생 BP)
 ├── AMS_Enemy_Rusher     — 돌진형 (파생 BP)
 └── AMS_Enemy_Shooter    — 원거리형 (파생 BP)
      ├── UHealthComponent       (플레이어와 동일 컴포넌트)
      └── UEnemyBehaviorComponent (교체 가능: 추적/돌진/원거리)

UDataAsset (x3)
 ├── UWeaponDataAsset     — 무기 스탯 (데미지, 쿨다운, 범위)
 ├── UEnemyDataAsset      — 적 스탯 (HP, 이속, 공격력)
 └── UWaveDataAsset       — 웨이브 구성 (적 종류, 수, 간격)

UI
 ├── WBP_HUD              — HP 바, 경험치 바, 타이머, 킬 수
 ├── WBP_LevelUpSelect    — 레벨업 시 무기/패시브 선택 팝업
 └── WBP_MainMenu         — 메인 메뉴 + 모드 선택
```

### 3.2 핵심 설계 원칙

1. **행동 = 컴포넌트, 데이터 = DataAsset**: 어떻게 동작할지는 컴포넌트가, 수치가 얼마인지는 DataAsset이 결정. 두 축이 독립적으로 교체 가능
2. **규칙 = GameMode, 상태 = GameState**: 게임 규칙 변경은 GameMode 교체만으로, 전역 상태는 GameState를 통해 누구나 접근
3. **이벤트 통신 = 델리게이트**: 체력 변경, 사망, 레벨업 등 오브젝트 간 통신은 직접 참조 대신 델리게이트로 분리
4. **C++ 기반 → BP 확장**: 핵심 시스템은 C++로, 세부 조정과 UI는 Blueprint로 구현

## 4. 주차별 제작 계획

### 4주차 (4/27) — 기본 서바이벌 루프 구축

**목표**: 플레이어가 움직이고, 적이 따라오고, HP가 줄어드는 최소 루프 완성

#### 제작 항목

| 번호 | 항목 | 분류 | 비고 |
|---|---|---|---|
| 4-1 | 프로젝트 생성 및 폴더 구조 설정 | 환경 | Source/, Content/ 하위 구조 확립 |
| 4-2 | `AMS_PlayerCharacter` C++ 클래스 생성 | C++ | ACharacter 상속 |
| 4-3 | Enhanced Input System으로 WASD 이동 구현 | C++ | `InputMappingContext`, `InputAction` 설정 |
| 4-4 | 탑다운 스프링 암 + 카메라 설정 | BP | 에디터에서 조정 |
| 4-5 | `AMS_PlayerState` C++ 클래스 생성 | C++ | HP, 경험치, 레벨 변수 (`UPROPERTY`) |
| 4-6 | `AMS_GM_Survival` 기본 GameMode 생성 | C++ | DefaultPawnClass, PlayerStateClass 지정 |
| 4-7 | `WBP_HUD` 기본 위젯 제작 | BP/UMG | HP 바, 경험치 바, 타이머 |
| 4-8 | 기본 적 `AMS_Enemy_Chaser` 생성 (스피어 메시) | BP | Tick 기반 플레이어 추적 이동 |
| 4-9 | 적과 충돌 시 HP 감소 처리 | C++ | `OnComponentHit` 또는 Overlap |
| 4-10 | HUD에 HP 바인딩 (PlayerState → HUD) | BP | `GetPlayerState` 활용 |

**4주차 완료 기준**: 플레이어가 WASD로 이동하고, 적이 따라오며, 충돌 시 HP 바가 줄어드는 화면이 동작함

---

### 5주차 (5/4) — C++/BP 무기 시스템 + 적 클래스 계층

**목표**: 무기 컴포넌트 구현으로 "컴포넌트 교체 = 동작 변경" 첫 체험

#### 제작 항목

| 번호 | 항목 | 분류 | 비고 |
|---|---|---|---|
| 5-1 | `UWeaponComponent` 추상 기본 클래스 (C++) | C++ | `UPROPERTY`, `UFUNCTION`, `virtual void Attack()` |
| 5-2 | `UMeleeWeaponComp` 파생: 회전 근접 공격 | C++ | 캐릭터 주위를 회전하며 범위 내 적 타격 |
| 5-3 | `URangedWeaponComp` 파생: 투사체 원거리 공격 | C++ | 가장 가까운 적에게 `AProjectile` 발사 |
| 5-4 | `UAOEWeaponComp` 파생: 범위 오라 공격 | C++ | 주기적으로 주변 반경 데미지 |
| 5-5 | BP로 `UMeleeWeaponComp` 파생 무기 하나 추가 | BP | C++와 동일 기능을 BP로 구현해 비교 |
| 5-6 | `AMS_EnemyBase` C++ 기반 클래스 작성 | C++ | HP, 이속 등 공통 속성 |
| 5-7 | BP 파생으로 `AMS_Enemy_Rusher`, `AMS_Enemy_Shooter` 추가 | BP | 색상·행동 구분 |
| 5-8 | PlayerCharacter에 `UWeaponComponent` 장착 확인 | 통합 | 에디터에서 컴포넌트 교체 실험 |

**5주차 완료 기준**: 에디터에서 `UWeaponComponent`를 다른 파생으로 교체하면 공격 방식이 달라지는 것 확인. 적이 3종으로 다양해짐

**핵심 체험 포인트**: "컴포넌트를 갈아끼우는 것만으로 행동이 변한다" — 상속 vs 컴포넌트 비교 토의

### 6주차 (5/11) — 델리게이트 통신 + 컴포넌트 완전 분리

**목표**: 오브젝트 간 직접 참조를 제거하고 델리게이트로 대체. 게임 핵심 루프 완성

#### 제작 항목

| 번호 | 항목 | 분류 | 비고 |
|---|---|---|---|
| 6-1 | `UHealthComponent` C++ 분리 | C++ | 플레이어·적 모두에 부착 가능한 공용 컴포넌트 |
| 6-2 | `FOnHealthChanged` 멀티캐스트 델리게이트 선언 | C++ | HP 변경 시 브로드캐스트 |
| 6-3 | HUD가 `OnHealthChanged`를 구독해 자동 갱신 | C++/BP | PlayerCharacter → HUD 직접 참조 제거 |
| 6-4 | `FOnEnemyDied` 델리게이트 → 경험치 지급 로직 | C++ | 적 사망 → PlayerState 경험치 증가 |
| 6-5 | 경험치 아이템(보석) 드롭 + Overlap 획득 | C++/BP | `AMS_ExpGem` 액터, Sphere Collision |
| 6-6 | 레벨업 시 `WBP_LevelUpSelect` 팝업 표시 | BP/UMG | 무기/패시브 3가지 선택지 |
| 6-7 | `UPassiveBuffComponent` 구현 | C++ | 이동속도 배율, 공격력 배율 등 |
| 6-8 | 런타임 컴포넌트 부착/제거 실험 | C++ | `AddComponent` / `DestroyComponent` |
| 6-9 | `UEnemyBehaviorComponent` 기반 클래스 | C++ | 추적/돌진/원거리를 파생으로 구현 |

**6주차 완료 기준**: 적을 죽이면 보석이 떨어지고, 레벨업하면 무기/패시브를 선택하고, 점점 강해지는 서바이벌 루프가 완성됨. 모든 통신이 델리게이트 기반

**핵심 체험 포인트**: `UHealthComponent`를 적에게 붙여 "플레이어와 적이 동일한 체력 컴포넌트를 공유"하는 것 확인. "왜 컴포넌트로 나눠야 하나?" 질문에 직접 답하는 순간

### 7주차 (5/18) — DataAsset + Data-Driven 전환

**목표**: 하드코딩된 수치를 DataAsset으로 분리. 코드 수정 없이 에디터에서 게임 밸런싱

#### 제작 항목

| 번호 | 항목 | 분류 | 비고 |
|---|---|---|---|
| 7-1 | `UWeaponDataAsset` 생성 (C++) | C++ | 데미지, 쿨다운, 범위, 메시 레퍼런스 |
| 7-2 | `UEnemyDataAsset` 생성 (C++) | C++ | HP, 이속, 공격력, 행동 컴포넌트 클래스 |
| 7-3 | `UWaveDataAsset` 생성 (C++) | C++ | 웨이브별 적 종류, 수, 스폰 간격 |
| 7-4 | `WeaponComponent`가 `UWeaponDataAsset`을 참조하도록 리팩토링 | C++ | 하드코딩 수치 → DataAsset 필드 |
| 7-5 | `EnemyBase`가 `UEnemyDataAsset`을 참조하도록 리팩토링 | C++ | 동일 패턴 적용 |
| 7-6 | 웨이브 스폰 시스템 (`AMS_WaveSpawner`) | C++ | `UWaveDataAsset` 기반으로 적 스폰 |
| 7-7 | `TSoftObjectPtr`로 무기 메시 비동기 로딩 | C++ | SoftPtr 체험 |
| 7-8 | 에디터에서 DataAsset 추가 → 게임 내 자동 등장 확인 | 통합 | "코드 한 줄 안 고치고 신규 무기 추가" 체험 |

**7주차 완료 기준**: 에디터에서 DataAsset을 새로 만들어 배열에 추가하면 게임 내 레벨업 선택지에 새 무기가 자동으로 나타남. 적 웨이브도 DataAsset으로 구성됨

**핵심 체험 포인트**: "왜 DataAsset인가?" — 기획자가 코드를 건드리지 않고 게임을 조정할 수 있는 구조

### 8주차 (5/25) — GameMode 전환 + 프로젝트 마무리

**목표**: GameMode 3개로 3가지 게임 모드 구현. 코딩 컨벤션 점검 후 최종 발표

#### 제작 항목

| 번호 | 항목 | 분류 | 비고 |
|---|---|---|---|
| 8-1 | `AMS_GM_TimeAttack` GameMode 구현 | C++ | 제한 시간(3분), 킬 수 최대화 |
| 8-2 | `AMS_GM_BossRush` GameMode 구현 | C++ | 보스 연속 출현, 클리어 수 기록 (여유 시 도전) |
| 8-3 | `AMS_GameState` 확장 — 모드별 상태 통합 관리 | C++ | 시간, 킬 수, 웨이브, 보스 번호 |
| 8-4 | `WBP_MainMenu` — 모드 선택 메뉴 | BP/UMG | 서바이벌/타임어택/보스러시 버튼 |
| 8-5 | 결과 화면 (`WBP_ResultScreen`) | BP/UMG | 최종 킬 수, 생존 시간, 리플레이 버튼 |
| 8-6 | 메뉴 → 모드 선택 → 플레이 → 결과 전체 흐름 연결 | 통합 | Level Streaming 또는 OpenLevel 활용 |
| 8-7 | 코딩 컨벤션 점검 + 리팩토링 | 검토 | 언리얼 네이밍 규칙(A/U/F 접두어), 폴더 정리 |
| 8-8 | 팀/개인 프로젝트 방향 논의 | 토의 | 이 구조를 기반으로 확장 방향 |

**8주차 완료 기준**: 메뉴에서 모드를 고르면 같은 맵에서 전혀 다른 규칙의 게임이 시작됨. "3개의 게임을 만든 게 아니라 GameMode 3개를 만든 것이다"를 체감

**핵심 체험 포인트**: GameMode 교체만으로 게임이 달라지는 것을 보여주며 2주차 이론을 실습으로 완결

## 6. 개발 우선순위 및 리스크 관리

### 우선순위 (Must → Should → Could)

| 우선순위 | 항목 |
|---|---|
| **Must** | PlayerCharacter WASD 이동, HealthComponent, 기본 적 추적 AI, HUD (HP/EXP), 레벨업 루프 |
| **Must** | WeaponComponent (근접 1종 이상), DataAsset 연동, GM_Survival |
| **Should** | 무기 3종 (근접/원거리/범위), 적 3종, GM_TimeAttack, 메인 메뉴 |
| **Could** | GM_BossRush, 보스 액터, 결과 화면, 코드 컨벤션 리팩토링 |

### 리스크 및 대응

| 리스크 | 대응 방안 |
|---|---|
| 수강생 C++ 빌드 환경 문제 | 4/27 전 환경 확인 공지. 빌드 오류 수강생은 BP 전용으로 진행 가능하도록 BP 파생 버전 병행 준비 |
| 구현 속도 편차 | Must 항목만으로도 매주 플레이 가능한 상태를 유지. Should/Could는 선택적으로 추가 |
| 델리게이트 이해 어려움 | 6주차 실습 전, "직접 참조 → 깨지는 예시"를 먼저 보여준 후 델리게이트로 해결하는 흐름으로 진행 |
| 컴파일 시간 | 핫 리로드 활용. 자주 수정되는 클래스는 BP 파생으로 분리해 재컴파일 최소화 |
