## 2. 1~3주차 실제 진행 분석

### 1주차 (3/23) — 언리얼 엔진 소개

| 구분 | 내용 |
|------|------|
| **원본 계획** | 언리얼 엔진의 기초, 기본 문법, 상속 |
| **실제 진행** | 엔진 소개 및 동기부여 중심의 오리엔테이션 |

**다룬 내용:**
- 왜 언리얼을 쓰는가? — C++ Native, 블루프린트, 오픈소스, 아티스트 친화적 생태계
- 유니티 vs 언리얼 설계 철학 차이 (자율적 구성 vs 통일된 프레임워크)
- 모던 객체지향 개념: 메모리 자동 관리, 상속보다 합성, 인터페이스 기반 설계, 리플렉션
- 강사 핵심 메시지: "엔진을 블랙박스로 보지 마라", "무엇이든 하나 만들어 보라"

**원본 대비 차이:** C++ 기본 문법/상속 실습 미진행. 엔진 철학과 "왜"에 집중.

### 2주차 (3/30) — GameMode / GameState

| 구분 | 내용 |
|------|------|
| **원본 계획** | 월드 생성 과정, GameMode/GameState, 라이프사이클 |
| **실제 진행** | 원본을 대체로 따르되, 멀티플레이 관점에서 심도 있게 진행 |

**다룬 내용:**

**원본 대비 차이:** 멀티플레이 관점이 원본보다 훨씬 깊음.

### 3주차 (4/6) — Actor 시스템과 캐릭터

| 구분 | 내용 |
|------|------|
| **원본 계획** | AActor/UObject, Pawn/Character, PlayerCharacter/PlayerState (실습) |
| **실제 진행** | UObject → AActor → APawn → ACharacter 계층 전체 + Pawn vs Character 실습 |

**다룬 내용:**
- UObject: 언리얼의 기본 단위, NewObject() 생성 과정(이름 결정 → 엔진 등록 → GC 등록)
- AActor: 월드에 종속된 오브젝트, Transform 보유, ULevel이 관리
- APawn: 플레이어 입력을 받을 수 있는 AActor, Possess(빙의) 개념
- ACharacter: 인간형 유닛, CharacterMovementComponent(CMC)
- **실습**: BP_JustPawn vs BP_ThirdPersonCharacter 직접 조작 비교
- 강사 정리: "CMC는 사세요(직접 구현 X)"

**미진행:** PlayerState, PlayerController 상세 (간략 언급만)

**강사가 명시한 "아직 다루지 않은 것":**
1. UI를 만드는 법
2. 객체들끼리 서로 통신하게 하는 법
3. 새로운 기능을 유지보수 가능하게 만드는 방법
4. 게임의 규모를 키우는 방법

## 3. 종합 평가

### 잘 된 점
- **"왜 이렇게 설계했는가"**에 집중하는 깊이 있는 이론 수업
- 2주차 멀티플레이 구조 설명이 원본 계획보다 풍부하고 실전적
- UObject → ACharacter 계층을 한 번에 관통하는 3주차 구성이 효과적
- 매 주차 "아직 다루지 않은 것"을 명시하여 학습 로드맵 가시성 유지

### 보완이 필요한 점
- **실습 비중 부족**: Pawn vs Character 비교 외에는 핸즈온 실습이 거의 없음
- **PlayerController / PlayerState 미진행**: 원본 3주차에 포함되어 있었으나 시간 부족으로 넘어감
- **C++ 기본 문법**: 1주차에서 생략됨
- 강사 본인이 "이론 수업만으로는 흥미 유발이나 실제 체험 제공이 어렵다"는 인식 보유

### 현재 프로젝트 상태
- ThirdPerson 템플릿 기반 프로젝트 (Week3.uproject, UE 5.4)
- Blueprint: `BP_Gamemode`, `BP_JustPawn`, `BP_ThirdPersonCharacter`
- Enhanced Input 구성 완료: `IMC_Default`, `IA_Jump`, `IA_Look`, `IA_Move`
- Mannequin 캐릭터(Manny/Quinn) 메시, 애니메이션 세팅 완비
- C++ 클래스: `AWeek3Character`, `AWeek3GameMode`
- → 4주차부터 이 프로젝트를 그대로 확장하며 실습 진행 가능

1. **PlayerController/PlayerState 누락**: 3주차에서 시간 부족으로 미진행. 입력 시스템의 핵심인 PlayerController를 별도로 다루지 않으면 이후 Enhanced Input 활용에 공백 발생

## 6. 주차별 상세 계획

### 4주차 (4/27) — PlayerController & Enhanced Input 심화

**변경 이유**: 3주차에서 Pawn/Character를 다뤘지만 PlayerController와 입력 시스템 상세는 미진행. "조작할 수 없으면 UI도 의미가 없다" — 입력을 먼저 다루는 것이 학습 흐름상 자연스러움.

**이론 (50분)**:
- **PlayerController의 역할과 Possess 메커니즘 상세**
  - 3주차 "빙의(Possess)" 개념의 Controller 관점 설명
  - APlayerController vs AAIController: 역할 분담
  - Controller → Pawn 관계의 설계 의도 (분리의 이유)
- **PlayerState: 플레이어별 상태 관리** (3주차 미진행분 보충)
  - 2주차 GameState(전체 상태) vs PlayerState(개별 상태) 대비
  - 멀티플레이에서 PlayerState의 복제 역할
- **Enhanced Input 시스템 심화**
  - 현재 프로젝트의 `IMC_Default`, `IA_Jump/Look/Move` 구조 분석
  - InputAction, InputMappingContext, Modifiers/Triggers 아키텍처
  - 기존 Input 시스템과의 차이점 및 데이터 주도 설계의 장점

**실습 (40분)**:
- 기존 BP_ThirdPersonCharacter를 확장
  - `IA_Sprint` 추가: Shift 키로 달리기 (CMC의 MaxWalkSpeed 변경)
  - `IA_Interact` 추가: E키 상호작용 입력 (7주차 델리게이트와 연결 예정)
  - Modifier/Trigger 조합 실험: Hold, Tap, Chorded Action 차이 체험
- **목표**: Enhanced Input의 데이터 주도 구조를 직접 체감

**연결 고리**: 3주차 APawn의 Possess → 4주차 PlayerController가 어떻게 입력을 전달하는가

---

### 5주차 (5/4) — HUD, 위젯 블루프린트, 블루프린트 기초

**변경 이유**: 원본 4/27 내용을 1주 뒤로 이동. 4주차에서 캐릭터 조작을 확장했으므로, 그 위에 UI를 올리는 것이 자연스러움.

**이론 (50분)**:
- **UMG(Unreal Motion Graphics) 기초**
  - 위젯 계층 구조, 앵커 시스템, 슬롯
  - 위젯 블루프린트 에디터: Designer 탭 vs Graph 탭
- **블루프린트 노드 기초**
  - 변수, 이벤트 그래프, 함수, 매크로
  - 블루프린트만으로 할 수 있는 것 vs C++이 필요한 것
- **C++ ↔ 블루프린트 연동**
  - `BlueprintCallable`, `BlueprintImplementableEvent`, `BlueprintNativeEvent`
  - 1주차 "블루프린트 = C++과 동일한 결과" 개념의 실전 적용
- **3주차 "UI를 만드는 법" 해결**

**실습 (40분)**:
- `WBP_HUD` 위젯 블루프린트 제작
  - 체력바(ProgressBar) + 스태미나 게이지
  - 4주차 Sprint와 연계: 달리면 스태미나 감소 → HUD에 실시간 반영
- PlayerState 값 → 위젯 바인딩: Polling vs Event-driven 두 방식 비교
- 간단한 블루프린트 로직: 체력 0 이하 시 Game Over 텍스트 표시

**연결 고리**: 4주차 Sprint 기능 → 5주차 스태미나 게이지 UI → 눈에 보이는 피드백

### 6주차 (5/11) — 언리얼 C++ 심화: GC, 리플렉션, 코딩 컨벤션

**변경 이유**: 4~5주차에서 블루프린트와 UPROPERTY를 이미 사용한 뒤이므로, "왜 이 매크로가 필요한가"를 체감한 상태에서 원리를 배우는 것이 효과적.

**이론 (50분)**:
- **가비지 콜렉터 동작 원리 심화**
  - 3주차 UObject 생성 시 GC 등록 → 내부 동작(Mark & Sweep, Cluster) 상세
  - `UPROPERTY()`를 안 붙이면 GC가 수거하는 이유 — 실제 크래시 시나리오
- **리플렉션 시스템**
  - `UPROPERTY`, `UFUNCTION`, `UCLASS`, `USTRUCT` 매크로의 역할
  - 에디터 노출, 직렬화, 네트워크 복제 — 모두 리플렉션 기반
  - 1주차 "리플렉션을 통한 객체 관리"의 구체적 구현
- **언리얼 코딩 컨벤션**
  - 접두사: `A`(Actor), `U`(UObject), `F`(구조체), `E`(Enum), `I`(Interface), `T`(Template)
  - Epic Games 공식 코딩 표준 핵심 요약

**실습 (40분)**:
- 4~5주차에서 작성한 코드를 컨벤션에 맞게 리팩토링
- `UPROPERTY` 스펙파이어 실험
  - `EditAnywhere` vs `VisibleAnywhere`, `BlueprintReadWrite` vs `BlueprintReadOnly`
  - `Category`, `meta=(ClampMin, ClampMax)` 등 에디터 UX 개선
- 의도적으로 `UPROPERTY()` 누락 후 GC 수거 관찰 (디버그 빌드에서 확인)

**연결 고리**: 5주차 블루프린트에서 사용한 변수들 → 6주차 "왜 이 매크로를 붙여야 했는가" 정리

---

### 7주차 (5/18) — 델리게이트, 컴포넌트 기반 설계

**변경 이유**: 3주차 말미 "객체 간 통신"과 "유지보수 가능한 설계" 두 가지 미다룬 주제를 동시에 해결.

**이론 (50분)**:
- **델리게이트(Delegate)**
  - 싱글캐스트, 멀티캐스트, 다이나믹 델리게이트
  - `DECLARE_DELEGATE`, `DECLARE_MULTICAST_DELEGATE`, `DECLARE_DYNAMIC_MULTICAST_DELEGATE`
  - 객체 간 느슨한 결합 — 2주차 GameMode/GameState 간 통신의 연장선
  - vs 인터페이스: 언제 델리게이트, 언제 인터페이스?
- **컴포넌트 기반 설계**
  - ActorComponent vs SceneComponent
  - 1주차 "합성 > 상속" 원칙의 실제 코드 적용
  - 예시: HealthComponent, InteractionComponent
- **3주차 "객체 간 통신"과 "유지보수 가능한 설계" 해결**

**실습 (40분)**:
- `UInteractionComponent` 제작
  - 4주차 E키 입력 → 7주차 델리게이트 이벤트 전파 연결
  - 문 열기(`AInteractableDoor`), 아이템 줍기(`APickupItem`)
  - `OnInteract` 멀티캐스트 델리게이트로 반응
- 5주차 HUD에 상호작용 프롬프트("E키를 눌러 상호작용") 추가
- 기존 체력 시스템을 `UHealthComponent`로 리팩토링

**연결 고리**: 4주차 E키 입력 → 7주차 상호작용 시스템으로 완성. 컴포넌트 분리 → 재사용성 체감

---

### 8주차 (5/25) — 에셋 관리 + 데이터 중심 설계 (통합)

**변경 이유**: 원본 2주분(5/18 포인터 타입, 5/25 DataAsset)을 1주로 통합. 초보 대상이므로 포인터 타입 상세보다 DataAsset/DataTable 중심의 실전적 내용에 비중. 확보한 1주를 9주차 종합 실습에 할당.

**이론 (50분)**:
- **에셋 참조 방식 핵심 정리**
  - `TObjectPtr`: 에디터 시간 참조 (기존 raw 포인터 대체)
  - `TSoftObjectPtr` / `TSoftClassPtr`: 비동기 로딩, 메모리 절약
  - `TSubclassOf<T>`: 클래스 타입 제한
  - Hard vs Soft Reference: 실전 판단 기준
- **DataAsset 활용 데이터 중심 설계**
  - 코드 하드코딩 → DataAsset 분리 패턴
  - `UPrimaryDataAsset` vs `UDataAsset`
- **DataTable**: CSV/JSON 기반 대량 데이터 관리
- **PrimaryAsset & AssetManager** 개념 소개 (간략)

**실습 (40분)**:
- `UCharacterDataAsset` 제작: 체력, 이동속도, 스태미나 등 스탯 외부화
  - 5주차 HUD 체력바가 DataAsset에서 값을 읽도록 리팩토링
- `FItemData` DataTable 제작: 아이템 이름, 아이콘, 효과를 테이블로 관리
  - 7주차 상호작용 시스템의 APickupItem이 DataTable에서 데이터를 읽도록 연결
- Soft Reference로 큰 에셋(메시, 텍스처) 참조 실험

**연결 고리**: 7주차 아이템 시스템 → 8주차 데이터 외부화 → "값 바꾸려면 코드가 아닌 에셋만 수정" 체감

---

### 9주차 (6/1) — 종합 실습 프로젝트 & 마무리

**변경 이유**: 원본 기말고사 휴강을 실습으로 전환. 1주차 강사의 "무엇이든 하나 만들어 보세요 — 어떤 강의도 직접 만드는 경험보다 더 많이 배울 순 없습니다"를 실현.

**내용 (90분 전체 실습)**:
- 4~8주차 누적 프로젝트 기반 미니 게임 완성
  - 캐릭터 이동+달리기(4주차) + HUD(5주차) + 상호작용(7주차) + 데이터(8주차)
  - 부족한 부분 보충 및 자유 확장 시간
- 게임 규모 확장 시 고려사항 (간략 15분)
  - 레벨 스트리밍, 서브시스템, 모듈 구조화
  - **3주차 "게임의 규모를 키우는 방법" 해결**
- 이후 학습 로드맵 안내
  - 애니메이션 시스템 (AnimBP, Animation Montage)
  - AI (Behavior Tree, Blackboard)
  - 네트워크 심화 (Replication 실전)
  - GAS (Gameplay Ability System)
- Q&A 및 피드백

## 7. 누적 프로젝트 흐름도

```
[Week3 프로젝트] BP_ThirdPersonCharacter + BP_Gamemode
    │
    ▼
4주차: + IA_Sprint, IA_Interact 입력 확장
    │     + PlayerState 기본 구성
    ▼
5주차: + WBP_HUD (체력바, 스태미나 게이지)
    │     + 블루프린트 ↔ C++ 연동
    ▼
6주차: 코드 리팩토링 (컨벤션 적용, UPROPERTY 정리)
    │
    ▼
7주차: + UInteractionComponent, UHealthComponent
    │     + 상호작용 오브젝트 (문, 아이템)
    │     + 델리게이트 이벤트 시스템
    ▼
8주차: + UCharacterDataAsset (스탯 외부화)
    │     + FItemData DataTable (아이템 데이터)
    │     + Soft Reference 적용
    ▼
9주차: → 미니 게임 완성 + 마무리
```

**장점:**
- 매주 **눈에 보이는 진전**을 경험
- 이전 주차 내용이 자연스럽게 복습됨
- 9주차에 별도의 "종합 프로젝트"를 새로 시작할 필요 없음

## 10. 교수법 제안

### 10-1. 이론-실습 비율 전환

| 구간 | 이론 | 실습 | 비고 |
|------|------|------|------|
| 1~3주차 (완료) | ~85분 | ~5분 | 기반 개념 집중 |
| 4~8주차 (향후) | ~50분 | ~40분 | 강사 의지 반영 |
| 9주차 (마무리) | ~15분 | ~75분 | 전체 실습 |

### 10-2. 미니 프로젝트 사전 공개

9주차 종합 실습의 **완성 목표를 4주차에 미리 공개**:

> "이번 학기가 끝날 때, 여러분은 캐릭터를 조작하고(4주차), 화면에 상태를 표시하고(5주차), 월드의 물체와 상호작용하고(7주차), 데이터를 외부에서 관리하는(8주차) 미니 게임을 완성하게 됩니다."

매주 "내가 배운 것이 최종 결과물의 어느 부분인지" 인식하며 학습 가능.

### 10-3. C++ 기초 보충 방안

1주차에서 생략된 "기본 문법, 상속" 실습에 대해:

- **학생 숙련도 충분**: 현재 계획대로 진행 (6주차에서 언리얼 특수 문법만)
- **보충 필요**: 중간고사 휴강 기간(4/13~4/20)에 자습 자료 배포 권장
  - 포인터, 참조, 상속, 가상함수, 스마트 포인터 기초
  - 4주차 시작 전에 10분 퀵 리뷰

### 10-4. 프로젝트 기반 평가 기준 (선택)

| 항목 | 배점 | 기준 |
|------|------|------|
| 캐릭터 조작 | 20% | Enhanced Input 기반 이동+달리기+상호작용 |
| UI/HUD | 20% | 체력/스태미나 표시, 상호작용 프롬프트 |
| 코드 품질 | 20% | 언리얼 코딩 컨벤션 준수, UPROPERTY 적절 사용 |
| 설계 구조 | 20% | 컴포넌트 분리, 델리게이트 활용 |
| 데이터 관리 | 20% | DataAsset/DataTable 사용, 하드코딩 최소화 |

### 10-5. 주차별 사전 과제 (선택)

매주 다음 주차 내용에 대한 짧은 예습 과제를 제시하면 90분 수업 시간을 더 효율적으로 활용 가능:
- 4주차 전: "프로젝트 열어서 IMC_Default, IA_Jump 에셋 열어보기"
- 5주차 전: "에디터에서 위젯 블루프린트 하나 만들어보기 (빈 화면 OK)"
- 6주차 전: "Week3Character.h의 UPROPERTY 매크로 의미 조사해오기"
- 7주차 전: "C++ delegate 패턴 간단히 검색해오기"
- 8주차 전: "DataAsset이 무엇인지 공식 문서 1페이지만 읽어오기"
