## 2. 1~3주차 실제 진행 분석

### 1주차 (3/23) — 언리얼 엔진 소개

| 구분 | 내용 |
|------|------|
| **원본 계획** | 언리얼 엔진의 기초, 기본 문법, 상속 |
| **실제 진행** | 엔진 소개 및 동기부여 중심 |

**다룬 내용:**
- 왜 언리얼을 쓰는가? — C++ Native, 블루프린트, 오픈소스, 아티스트 친화적 생태계
- 유니티 vs 언리얼 설계 철학 차이 (자율적 구성 vs 통일된 프레임워크)
- 모던 객체지향 개념: 메모리 자동 관리, 상속보다 합성, 인터페이스 기반 설계, 리플렉션
- 강사 핵심 메시지: "엔진을 블랙박스로 보지 마라", "무엇이든 하나 만들어 보라"

**원본 대비 차이:** C++ 기본 문법/상속 실습 미진행. 엔진 철학과 "왜"에 집중한 오리엔테이션.

### 2주차 (3/30) — GameMode / GameState

| 구분 | 내용 |
|------|------|
| **원본 계획** | 월드 생성 과정, GameMode/GameState, 라이프사이클 |
| **실제 진행** | 원본을 대체로 따르되, 멀티플레이 관점에서 심도 있게 진행 |

**다룬 내용:**
- GameInstance → UWorld → GameMode 스폰 과정 (PIE 코드 기반 설명)
- GameMode = Modern GameManager: 게임 룰 관장, 매니저 클래스 지정 권한
- 멀티플레이 구조: 서버/클라이언트, RPC vs Replication
- GameState: 복제되는 상태 정보, 서버 Authority 개념
- "룰(GameMode, 서버 전용) / 상태(GameState, 공유)" 분리 원칙
- 강사: "실습을 조금 늘려보긴 할 것", "지루한 부분을 잘 해놔야 나중에 재밌는 걸 할 때 막힘이 없다"

**원본 대비 차이:** 멀티플레이 관점이 원본보다 훨씬 깊음. 라이프사이클 전체보다 GameMode/GameState에 집중.

### 3주차 (4/6) — Actor 시스템과 캐릭터

| 구분 | 내용 |
|------|------|
| **원본 계획** | AActor/UObject, Pawn/Character, PlayerCharacter/PlayerState (실습) |
| **실제 진행** | UObject → AActor → APawn → ACharacter 계층 전체 + Pawn vs Character 실습 |

**다룬 내용:**
- UObject: 언리얼의 기본 단위, NewObject() 생성 과정(이름 결정 → 엔진 등록 → GC 등록)
- AActor: 월드에 종속된 오브젝트, Transform 보유, ULevel이 관리
- APawn: 플레이어 입력을 받을 수 있는 AActor, Possess(빙의) 개념
- ACharacter: 인간형 유닛, CharacterMovementComponent(CMC)
- **실습**: BP_JustPawn vs BP_ThirdPersonCharacter 직접 조작 비교
- 강사 정리: "CMC는 사세요(직접 구현 X)"

**미진행:** PlayerState, PlayerController 상세 (간략 언급만)

**강사가 명시한 "아직 다루지 않은 것":**
1. UI를 만드는 법
2. 객체들끼리 서로 통신하게 하는 법
3. 새로운 기능을 유지보수 가능하게 만드는 방법
4. 게임의 규모를 키우는 방법

## 3. 종합 평가

### 잘 된 점
- **"왜 이렇게 설계했는가"**에 집중하는 깊이 있는 이론 수업
- 2주차 멀티플레이 구조 설명이 원본 계획보다 풍부하고 실전적
- UObject → ACharacter 계층을 한 번에 관통하는 3주차 구성이 효과적
- 매 주차 자료 마지막에 "아직 다루지 않은 것"을 명시하여 학습 로드맵 가시성 유지

### 보완이 필요한 점
- **실습 비중 부족**: Pawn vs Character 비교 외에는 핸즈온 실습이 거의 없음
- **PlayerController / PlayerState 미진행**: 원본 3주차에 포함되어 있었으나 시간 부족으로 넘어감
- **C++ 기본 문법**: 1주차에서 생략됨 — 학생 숙련도에 따라 보충 필요 여부 판단 필요
- 강사 본인이 "이론 수업만으로는 흥미 유발이나 실제 체험 제공이 어렵다"는 인식을 가지고 있음

### 현재 프로젝트 상태
- ThirdPerson 템플릿 기반 프로젝트 (Week3.uproject)
- `BP_Gamemode`, `BP_JustPawn`, `BP_ThirdPersonCharacter` 블루프린트 존재
- Enhanced Input 에셋 구성 완료: `IMC_Default`, `IA_Jump`, `IA_Look`, `IA_Move`
- Mannequin 캐릭터(Manny/Quinn) 메시, 애니메이션, 리그 세팅 완비
- → 4주차부터 이 프로젝트를 그대로 확장하며 실습 진행 가능

## 5. 주차별 상세 계획

### 4주차 (4/27) — PlayerController & Enhanced Input 심화 + 실습

**변경 이유**: 3주차에서 Pawn/Character를 다뤘지만 PlayerController와 입력 시스템 상세는 미진행. 원본의 HUD/블루프린트(4/27)보다 입력 처리를 먼저 다루는 것이 학습 흐름상 자연스러움 — 조작할 수 없으면 UI도 의미가 없다.

**이론 (50분)**:
- PlayerController의 역할과 Possess 메커니즘 상세
  - 3주차에서 Pawn의 "빙의" 개념을 배웠으므로, 그 반대편인 Controller 관점 설명
  - APlayerController vs AAIController 차이
- PlayerState: 플레이어별 상태 관리 (3주차 미진행분 보충)
  - 2주차 GameState(게임 전체 상태)와의 대비: PlayerState는 개별 플레이어 상태
- Enhanced Input 시스템 심화
  - 현재 프로젝트의 `IMC_Default`, `IA_Jump/Look/Move` 구조 분석
  - InputAction, InputMappingContext, Modifiers/Triggers 구조
  - 기존 Input 시스템과의 차이점 및 마이그레이션 이유

**실습 (40분)**:
- 기존 Week3 프로젝트의 BP_ThirdPersonCharacter를 확장
  - `IA_Sprint` 추가: 달리기(CMC의 MaxWalkSpeed 변경)
  - `IA_Interact` 추가: E키 상호작용 입력 (7주차 델리게이트와 연결 예정)
  - Modifier/Trigger 조합 실험: Hold, Tap, Chorded Action
- 목표: Enhanced Input의 데이터 주도 구조를 직접 체감

### 5주차 (5/4) — HUD, 위젯 블루프린트, 블루프린트 기초 + 실습

**변경 이유**: 원본 4/27 내용을 한 주 뒤로 이동. "입력 → UI" 순서가 논리적 — 4주차에서 만든 캐릭터 조작 위에 UI를 올린다.

**이론 (50분)**:
- UMG(Unreal Motion Graphics) 기초: 위젯 계층 구조, 앵커, 슬롯
- 위젯 블루프린트 에디터: Designer vs Graph
- 블루프린트 노드 기초: 변수, 이벤트 그래프, 함수, 매크로
- C++ ↔ 블루프린트 연동
  - `BlueprintCallable`, `BlueprintImplementableEvent`, `BlueprintNativeEvent`
  - 1주차에서 소개한 "블루프린트 = C++과 동일한 결과" 개념의 실전 적용
- **강사가 3주차에서 언급한 "UI를 만드는 법" 해결**

**실습 (40분)**:
- 체력바(ProgressBar) + 스태미나 게이지 HUD 제작
  - 4주차 Sprint 실습과 연계: 달리면 스태미나 감소 → HUD에 반영
- PlayerState의 값을 위젯에 바인딩하는 두 가지 방법 비교
  - Polling(Tick 기반) vs Event-driven(Property 변경 알림)
- 간단한 블루프린트 로직: 체력이 0 이하면 Game Over 텍스트 표시

### 6주차 (5/11) — 언리얼 C++ 심화: GC, 리플렉션, 코딩 컨벤션 + 실습

**변경 이유**: 4~5주차에서 블루프린트와 UPROPERTY를 이미 사용했으므로, "왜 이 매크로가 필요한가"를 체감한 상태에서 시스템 원리를 배우는 것이 효과적.

**이론 (50분)**:
- 가비지 콜렉터 동작 원리 심화
  - 3주차 UObject 생성 시 GC 등록 → 이제 내부 동작(Mark & Sweep, Cluster) 상세
  - `UPROPERTY()`를 안 붙이면 GC가 수거하는 이유 — 실제 크래시 시나리오
- 리플렉션 시스템: `UPROPERTY`, `UFUNCTION`, `UCLASS`, `USTRUCT` 매크로
  - 1주차 "리플렉션을 통한 객체 관리"의 구체적 구현
  - 에디터 노출, 직렬화, 네트워크 복제 — 모두 리플렉션 기반
- 언리얼 코딩 컨벤션
  - 접두사 규칙: `A`(Actor), `U`(UObject), `F`(구조체/일반), `E`(Enum), `I`(Interface), `T`(Template)
  - Epic Games 공식 코딩 표준 핵심 요약

**실습 (40분)**:
- 4~5주차에서 작성한 코드를 컨벤션에 맞게 리팩토링
- `UPROPERTY` 스펙파이어 실험
  - `EditAnywhere` vs `VisibleAnywhere`, `BlueprintReadWrite` vs `BlueprintReadOnly`
  - `Category`, `meta=(ClampMin, ClampMax)` 등 에디터 UX 개선
- 의도적으로 `UPROPERTY()` 누락 후 GC 수거 관찰 (디버그 빌드)

### 7주차 (5/18) — 설계 방법론: 델리게이트, 컴포넌트 기반 설계 + 실습

**변경 이유**: 3주차 말미에 "객체 간 통신"과 "유지보수 가능한 설계"가 미다룬 주제로 명시. 이 주차에서 두 가지를 모두 해결.

**이론 (50분)**:
- 델리게이트(Delegate)
  - 싱글캐스트, 멀티캐스트, 다이나믹 델리게이트
  - `DECLARE_DELEGATE`, `DECLARE_MULTICAST_DELEGATE`, `DECLARE_DYNAMIC_MULTICAST_DELEGATE`
  - 객체 간 느슨한 결합 — 2주차 GameMode/GameState 간 통신의 연장선
  - vs 인터페이스: 언제 델리게이트, 언제 인터페이스를 쓸 것인가
- 컴포넌트 기반 설계
  - ActorComponent vs SceneComponent
  - 1주차 "합성 > 상속" 원칙의 실제 코드 적용
  - 예시: HealthComponent, InteractionComponent
- **강사 언급 "객체 간 통신"과 "유지보수 가능한 설계" 동시 해결**

**실습 (40분)**:
- `UInteractionComponent` 제작: 4주차 E키 입력 → 7주차 델리게이트 이벤트 전파
  - 문 열기(AInteractableDoor), 아이템 줍기(APickupItem)
  - `OnInteract` 멀티캐스트 델리게이트로 반응
- 5주차 HUD에 상호작용 프롬프트("E키를 눌러 상호작용") 추가
- 기존 체력 시스템을 `UHealthComponent`로 리팩토링

### 8주차 (5/25) — 에셋 관리 + 데이터 중심 설계 (통합) + 실습

**변경 이유**: 원본 2주(5/18, 5/25)분을 1주로 통합. 포인터 타입 상세보다 DataAsset/DataTable 중심의 실전적 내용에 비중. 확보한 1주를 9주차 종합 실습에 할당.

**이론 (50분)**:
- 에셋 참조 방식 핵심 정리
  - `TObjectPtr`: 에디터 시간 참조 (기존 raw 포인터 대체)
  - `TSoftObjectPtr` / `TSoftClassPtr`: 비동기 로딩, 메모리 절약
  - `TSubclassOf<T>`: 클래스 타입 제한
  - Hard Reference vs Soft Reference: 메모리 로딩 차이와 실전 판단 기준
- DataAsset을 활용한 데이터 중심 설계
  - 코드에 하드코딩된 스탯값 → DataAsset으로 분리하는 패턴
  - UPrimaryDataAsset vs UDataAsset
- DataTable: CSV/JSON 기반 대량 데이터 관리
- PrimaryAsset & AssetManager 개념 소개 (간략)

**실습 (40분)**:
- `UCharacterDataAsset` 제작: 체력, 이동속도, 스태미나 등 캐릭터 스탯 외부화
  - 5주차 HUD 체력바가 DataAsset에서 값을 읽도록 리팩토링
- `FItemData` DataTable 제작: 아이템 이름, 아이콘, 효과를 테이블로 관리
  - 7주차 상호작용 시스템의 APickupItem이 DataTable에서 데이터를 읽도록 연결
- Soft Reference로 큰 에셋(메시, 텍스처) 참조 실험

### 9주차 (6/1) — 종합 실습 프로젝트 & 마무리

**변경 이유**: 원본 기말고사 휴강을 실습으로 전환. 1주차 강사의 "무엇이든 하나 만들어 보세요 — 어떤 강의도 직접 만드는 경험보다 더 많이 배울 순 없습니다"를 실현.

**내용 (90분 전체 실습)**:
- 4~8주차 누적 프로젝트 기반 미니 게임 완성
  - 캐릭터 이동+달리기(4주차) + HUD(5주차) + 상호작용(7주차) + 데이터(8주차)
  - 부족한 부분 보충 및 자유 확장 시간
- 게임 규모 확장 시 고려사항 (간략 15분 안내)
  - 레벨 스트리밍, 서브시스템, 모듈 구조화
  - **3주차 말미 "게임의 규모를 키우는 방법" 해결**
- 이후 학습 로드맵 안내
  - 애니메이션 시스템(ABP, Animation Montage)
  - AI (Behavior Tree, Blackboard)
  - 네트워크 심화 (Replication 실전)
  - GAS(Gameplay Ability System)
- Q&A 및 피드백

## 6. 누적 프로젝트 흐름도

4~9주차 실습이 하나의 프로젝트에 순차적으로 쌓이는 구조:

```
[Week3 프로젝트] BP_ThirdPersonCharacter + BP_Gamemode
    |
    v
4주차: + IA_Sprint, IA_Interact 입력 확장
    |     + PlayerState 기본 구성
    v
5주차: + WBP_HUD (체력바, 스태미나 게이지)
    |     + 블루프린트 ↔ C++ 연동
    v
6주차: 코드 리팩토링 (컨벤션 적용, UPROPERTY 정리)
    |
    v
7주차: + UInteractionComponent, UHealthComponent
    |     + 상호작용 오브젝트 (문, 아이템)
    |     + 델리게이트 이벤트 시스템
    v
8주차: + UCharacterDataAsset (스탯 외부화)
    |     + FItemData DataTable (아이템 데이터)
    |     + Soft Reference 적용
    v
9주차: → 미니 게임 완성 + 마무리
```

이 구조의 장점:
- 매주 **눈에 보이는 진전**을 경험
- 이전 주차 내용이 자연스럽게 복습됨
- 9주차에 별도의 "종합 프로젝트"를 새로 시작할 필요 없음

## 9. 교수법 제안

### 9-1. 이론-실습 비율 전환

| 구간 | 이론 | 실습 | 비고 |
|------|------|------|------|
| 1~3주차 (완료) | ~85분 | ~5분 | 기반 개념 집중 (자연스러운 흐름) |
| 4~8주차 (향후) | ~50분 | ~40분 | 강사 의지 + 학생 체감 균형 |
| 9주차 (마무리) | ~15분 | ~75분 | 전체 실습 |

### 9-2. 미니 프로젝트 사전 공개

9주차 종합 실습의 **완성 목표를 4주차에 미리 공개**합니다:

> "이번 학기가 끝날 때, 여러분은 캐릭터를 조작하고(4주차), 화면에 상태를 표시하고(5주차), 월드의 물체와 상호작용하고(7주차), 데이터를 외부에서 관리하는(8주차) 미니 게임을 완성하게 됩니다."

이렇게 하면 매주 "내가 배운 것이 최종 결과물의 어느 부분인지" 인식하며 학습 가능.

### 9-3. C++ 기초 보충 방안

1주차에서 생략된 "기본 문법, 상속" 실습에 대해:

- **학생 숙련도가 충분한 경우**: 현재 계획대로 진행 (6주차에서 언리얼 특수 문법만)
- **보충이 필요한 경우**: 중간고사 휴강 기간(4/13~4/20)에 자습 자료 배포 권장
  - 포인터, 참조, 상속, 가상함수, 스마트 포인터 기초
  - 4주차 시작 전에 10분 퀵 리뷰

### 9-4. 프로젝트 기반 평가 기준 (선택)

누적 프로젝트 방식을 채택할 경우, 9주차 최종 결과물의 평가 기준안:

| 항목 | 배점 | 기준 |
|------|------|------|
| 캐릭터 조작 | 20% | Enhanced Input 기반 이동+달리기+상호작용 |
| UI/HUD | 20% | 체력/스태미나 표시, 상호작용 프롬프트 |
| 코드 품질 | 20% | 언리얼 코딩 컨벤션 준수, UPROPERTY 적절 사용 |
| 설계 구조 | 20% | 컴포넌트 분리, 델리게이트 활용 |
| 데이터 관리 | 20% | DataAsset/DataTable 사용, 하드코딩 최소화 |
