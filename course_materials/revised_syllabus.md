## 2. 1~3주차 실제 진행 분석

### 1주차 (3/23) — 언리얼 엔진 소개

| 구분 | 내용 |
|------|------|
| **원본 계획** | 언리얼 엔진의 기초, 기본 문법, 상속 |
| **실제 진행** | 엔진 소개 및 동기부여 중심 |

**다룬 내용:**
- 왜 언리얼을 쓰는가? — C++ Native, 블루프린트, 오픈소스, 아티스트 친화적 생태계
- 유니티 vs 언리얼 설계 철학 차이 (자율적 구성 vs 통일된 프레임워크)
- 모던 객체지향 개념 소개: 메모리 자동 관리, 상속보다 합성, 인터페이스 기반 설계, 리플렉션
- 강사의 핵심 메시지: "엔진을 블랙박스로 보지 마라", "무엇이든 하나 만들어 보라"

**원본 대비 차이:** C++ 기본 문법/상속 실습은 진행하지 않음. 엔진 철학과 "왜"에 집중한 오리엔테이션 성격.

### 2주차 (3/30) — GameMode / GameState

| 구분 | 내용 |
|------|------|
| **원본 계획** | 월드 생성 과정, GameMode/GameState, 라이프사이클 |
| **실제 진행** | 원본 계획을 대체로 따르되, 멀티플레이 관점에서 심도 있게 진행 |

**다룬 내용:**
- GameInstance → UWorld → GameMode 스폰 과정 (PIE 코드 기반 설명)
- GameMode = Modern GameManager: 게임 룰 관장, 매니저 클래스 지정 권한
- 멀티플레이 구조: 서버/클라이언트, RPC vs Replication
- GameState: 복제(Replication)되는 상태 정보, 서버 Authority 개념
- "룰(GameMode, 서버 전용) / 상태(GameState, 공유)" 분리 원칙
- 강사 코멘트: "실습을 조금 늘려보긴 할 것", "지루한 부분을 잘 해놔야 나중에 재밌는 걸 할 때 막힘이 없다"

**원본 대비 차이:** 멀티플레이 관점의 GameMode/GameState 설명이 원본보다 훨씬 깊음. 라이프사이클 전체보다는 GameMode/GameState에 집중.

### 3주차 (4/6) — Actor 시스템과 캐릭터

| 구분 | 내용 |
|------|------|
| **원본 계획** | AActor/UObject, Pawn/Character, PlayerCharacter/PlayerState (실습) |
| **실제 진행** | UObject → AActor → APawn → ACharacter 상속 계층 전체 + Pawn vs Character 실습 |

**다룬 내용:**
- UObject: 언리얼의 기본 단위, 엔진 레벨 관리, NewObject() 생성 과정(이름 결정 → 엔진 등록 → GC 등록)
- AActor: 월드에 종속된 오브젝트, Transform(위치) 보유, ULevel이 관리
- APawn: 플레이어 입력을 받을 수 있는 AActor, Possess(빙의) 개념
- ACharacter: 인간형 유닛, CharacterMovementComponent(CMC)로 자연스러운 이동 구현
- **실습**: Pawn vs Character 직접 조작 비교
- 강사의 정리: "CMC는 사세요(직접 구현 X)"

**미진행 항목:** PlayerState, PlayerController 상세 (간략 언급만)

**강사가 직접 언급한 "아직 다루지 않은 것":**
1. UI를 만드는 법
2. 객체들끼리 서로 통신하게 하는 법
3. 새로운 기능을 유지보수 가능하게 만드는 방법
4. 게임의 규모를 키우는 방법

## 3. 종합 평가

### 잘 된 점
- **"왜 이렇게 설계했는가"**에 집중하는 깊이 있는 이론 수업
- 2주차 멀티플레이 구조 설명은 원본 계획보다 풍부하고 실전적
- UObject → ACharacter 계층을 한 번에 관통하는 3주차 구성이 효과적
- 매 주차 자료 마지막에 "아직 다루지 않은 것"을 명시하여 학습 로드맵 가시성 유지

### 보완이 필요한 점
- **실습 비중 부족**: 3주차 Pawn vs Character 비교 외에는 핸즈온 실습이 거의 없음
- **PlayerController / PlayerState 미진행**: 원본 3주차 계획에 포함되어 있었으나 시간 부족으로 넘어감
- **C++ 기본 문법**: 1주차에서 계획했으나 생략됨 — 학생들의 C++ 숙련도에 따라 보충 필요 여부 판단 필요
- 강사 본인이 "이론 수업만으로는 흥미 유발이나 실제 체험 제공이 어렵다"는 인식을 가지고 있음

## 5. 주차별 상세 계획

### 4주차 (4/27) — PlayerController & Enhanced Input 심화 + 실습

**변경 이유**: 3주차에서 Pawn/Character를 다뤘지만 PlayerController와 입력 시스템 상세는 미진행. 원본의 HUD/블루프린트(4/27)보다 **입력 처리를 먼저** 다루는 것이 학습 흐름상 자연스러움 — 조작할 수 없으면 UI도 의미가 없다.

**이론 (50분)**:
- PlayerController의 역할과 Possess 메커니즘 상세
  - 3주차에서 Pawn의 "빙의" 개념을 배웠으므로, 그 반대편인 Controller 관점 설명
- PlayerState: 플레이어별 상태 관리 (3주차 미진행분 보충)
  - 2주차 GameState(게임 전체 상태)와의 대비: PlayerState는 개별 플레이어 상태
- Enhanced Input 시스템 심화
  - InputAction, InputMappingContext, Modifiers/Triggers 구조
  - 기존 Input 시스템과의 차이점

**실습 (40분)**:
- 커스텀 입력 바인딩으로 캐릭터 조작 확장
  - 달리기(Sprint), 점프 높이 조절, 상호작용 키(E키) 등
  - 3주차 Pawn vs Character 실습의 Character를 이어서 확장

### 5주차 (5/4) — HUD, 위젯 블루프린트, 블루프린트 기초 + 실습

**변경 이유**: 원본 4/27 내용을 한 주 뒤로 이동. "입력 → UI" 순서가 논리적 — 4주차에서 만든 캐릭터 조작 위에 UI를 올린다.

**이론 (50분)**:
- UMG(Unreal Motion Graphics) 기초: 위젯 계층 구조
- 위젯 블루프린트로 HUD 만들기
- 블루프린트 노드 기초: 변수, 이벤트 그래프, 함수
- C++ ↔ 블루프린트 연동
  - `BlueprintCallable`, `BlueprintImplementableEvent`, `BlueprintNativeEvent`
  - 1주차에서 소개한 "블루프린트 = C++과 동일한 결과" 개념의 실전 적용

**실습 (40분)**:
- 4주차에서 만든 캐릭터에 체력바/스태미나 HUD 추가
- PlayerState의 값을 위젯에 바인딩
- **강사가 3주차에서 언급한 "UI를 만드는 법"을 이 주차에서 해결**

### 6주차 (5/11) — 언리얼 C++ 심화: GC, 리플렉션, 코딩 컨벤션 + 실습

**변경 이유**: 원본 내용 유지하되, 이전 주차들에서 이미 개념을 접한 GC·리플렉션을 체계적으로 정리하는 위치로 적합.

**이론 (50분)**:
- 가비지 콜렉터 동작 원리 심화
  - 2주차에서 개념 소개(참조 없으면 수거), 3주차에서 UObject 생성 시 GC 등록 확인 → 이제 내부 동작 상세
- 리플렉션 시스템: `UPROPERTY`, `UFUNCTION`, `UCLASS` 매크로
  - 1주차에서 소개한 "리플렉션을 통한 객체 관리"의 구체적 구현
- 언리얼 코딩 컨벤션: 네이밍 규칙(`A`, `U`, `F`, `E` 접두사), 매크로 사용 규칙

**실습 (40분)**:
- 기존 프로젝트 코드를 컨벤션에 맞게 리팩토링
- `UPROPERTY` 스펙파이어(`EditAnywhere`, `BlueprintReadWrite` 등) 실험

### 7주차 (5/18) — 설계 방법론: 델리게이트, 컴포넌트 기반 설계 + 실습

**변경 이유**: 3주차 말미에 "객체들끼리 서로 통신하게 하는 법"과 "유지보수 가능한 설계"가 미다룬 주제로 명시됨. 이 주차에서 두 가지를 모두 해결.

**이론 (50분)**:
- 델리게이트(Delegate)
  - 싱글캐스트, 멀티캐스트, 다이나믹 델리게이트
  - 객체 간 느슨한 결합(Loose Coupling) — 2주차 GameMode/GameState 간 통신의 연장선
- 컴포넌트 기반 설계
  - ActorComponent, SceneComponent
  - 1주차에서 소개한 "합성 > 상속" 원칙의 실제 코드 적용
  - 1주차 슬라이드의 "모델링만 다른 캐릭터" 예시를 컴포넌트로 해결

**실습 (40분)**:
- 상호작용 시스템 구현: 문 열기, 아이템 줍기
  - 델리게이트로 이벤트 전파 (캐릭터 → 문/아이템)
  - 4주차에서 만든 E키 상호작용 입력과 연결
- **강사 언급 "객체 간 통신"과 "유지보수 가능한 설계" 동시 해결**

### 8주차 (5/25) — 에셋 관리 + 데이터 중심 설계 (통합) + 실습

**변경 이유**: 원본에서 2주(5/18, 5/25)에 걸쳐 다루던 에셋 관리를 1주로 통합. 포인터 타입 상세보다 DataAsset/DataTable 중심의 실전적 내용에 비중을 둠. 이렇게 확보한 1주를 9주차 종합 실습에 할당.

**이론 (50분)**:
- 에셋 참조 방식 핵심 정리
  - `TObjectPtr`, `TSoftObjectPtr`, `TSubclassOf` — 각각 언제 쓰는지 비교
  - Hard Reference vs Soft Reference: 메모리 로딩 차이
- DataAsset을 활용한 데이터 중심 설계
  - 코드에 하드코딩된 값을 DataAsset으로 분리
- DataTable: 스프레드시트 기반 대량 데이터 관리
- PrimaryAsset 개념 소개 (간략)

**실습 (40분)**:
- DataAsset으로 캐릭터 스탯 시스템 구성
  - 5주차 HUD의 체력바가 DataAsset에서 값을 읽도록 리팩토링
- DataTable로 아이템 테이블 구성 (7주차 상호작용 시스템과 연계)

### 9주차 (6/1) — 종합 실습 프로젝트 & 마무리

**변경 이유**: 원본에서는 기말고사 휴강이었으나, 에셋 관리 2주분을 통합하여 확보한 1주를 활용. 1주차에서 강사가 강조한 "무엇이든 하나 만들어 보세요 — 어떤 강의도 직접 만드는 경험보다 더 많이 배울 순 없습니다"를 실현하는 주차.

**내용 (90분 전체 실습)**:
- 4~8주차에서 누적 구축한 프로젝트를 기반으로 미니 게임 완성
  - 캐릭터 이동(4주차) + HUD(5주차) + 상호작용(7주차) + 데이터(8주차)

## 7. 교수법 제안

### 7-1. 이론-실습 비율 조정

현재까지(1~3주차) 이론 중심이었던 수업을 4주차부터 **이론 50분 + 실습 40분** 구조로 전환합니다. 강사가 2주차에서 "실습을 조금 늘려보긴 할 것"이라고 언급한 점, 그리고 1주차에서 "무엇이든 하나 만들어 보세요"를 강조한 점을 모두 반영합니다.

### 7-2. 누적 프로젝트 방식

매주 실습을 독립적인 예제가 아닌, **하나의 프로젝트에 기능을 쌓아가는 방식**으로 운영합니다:

```
4주차: 캐릭터 + 입력 확장 (달리기, 상호작용)
  ↓ 이어서
5주차: + 체력/스태미나 HUD 추가
  ↓ 이어서
6주차: 코드 리팩토링 (컨벤션 적용)
  ↓ 이어서
7주차: + 상호작용 시스템 (문, 아이템)
  ↓ 이어서
8주차: + DataAsset으로 데이터 구조화
  ↓ 이어서
9주차: → 미니 게임 완성
```

이 구조의 장점:
- 학생이 매주 **눈에 보이는 진전**을 경험함
- 이전 주차 내용이 자연스럽게 복습됨
- 9주차에 별도의 "종합 프로젝트"를 새로 시작할 필요 없음

### 7-3. 3주차에서 언급된 4대 미다룬 주제 커버리지

| 미다룬 주제 (3주차 말미) | 해결 주차 |
|--------------------------|-----------|
| UI를 만드는 법 | 5주차 (HUD, 위젯 블루프린트) |
| 객체 간 통신 | 7주차 (델리게이트) |
| 유지보수 가능한 설계 | 7주차 (컴포넌트 기반 설계) |
| 게임 규모 확장 | 9주차 (종합 실습 내 간략 안내) |

### 7-4. C++ 기초 보충 여부

1주차에서 계획했던 "기본 문법, 상속" 실습이 생략되었습니다. 학생들의 C++ 숙련도에 따라:
- **숙련된 경우**: 현재 계획대로 진행 (6주차 C++ 심화에서 언리얼 특수 문법만 다룸)
- **미숙한 경우**: 4주차 시작 전에 C++ 기초 보충 자료(포인터, 상속, 가상함수)를 배포하고 자습을 유도하거나, 4주차 이론 시간 10분을 할애하여 핵심만 짚고 넘어가는 것을 권장

### 7-5. 미니 프로젝트 사전 공개

9주차 종합 실습에서 완성할 미니 프로젝트의 **요구사항을 4주차에 미리 공개**하면, 학생들이 매주 "내가 배운 것이 최종 프로젝트의 어떤 부분에 해당하는지" 인식하며 학습할 수 있습니다.
