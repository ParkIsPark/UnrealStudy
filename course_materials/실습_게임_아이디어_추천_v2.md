# 언리얼 스터디 — 실습용 게임 아이디어: 모듈형 서바이벌 액션

> 작성일: 2026-04-11  
> 기반: 강의 계획 수정본 v4 + 사용자 피드백 (GameMode 활용, 모듈화 설계 강조)

---

## 핵심 컨셉: "붙였다 떼는" 뱀서류

Vampire Survivors 스타일의 탑다운 서바이벌을 기본 뼈대로 하되, **기능 단위를 컴포넌트/모듈로 분리해서 붙였다 떼었다 할 수 있는 구조**로 설계한다. 이 구조의 목적은 두 가지다:

1. **언리얼 프레임워크의 설계 의도를 체감**: GameMode가 규칙을 관장하고, GameState가 상태를 공유하는 구조가 왜 존재하는지를 "게임 모드 전환"이라는 실체로 경험
2. **컴포넌트 기반 모듈화의 실질적 이점 체험**: 무기, 능력, 이동 방식 등을 컴포넌트 단위로 교체 가능하게 만들면 "같은 캐릭터가 전혀 다른 게임처럼 동작"하는 것을 직접 확인

---

## 게임 구조

### 기본 뼈대 (4~5주차에 완성)

```
플레이어(탑다운 이동) → 적 웨이브 스폰 → 처치 시 경험치 → 레벨업 → 무기/능력 선택 → 반복
```

이것만으로도 플레이 가능한 서바이벌 루프가 된다.

### 모듈 확장 (6~8주차에 기능을 붙였다 뗐다)

기본 뼈대 위에 **독립된 모듈**들을 컴포넌트로 만들어 추가/제거할 수 있도록 설계한다:

| 모듈 | 역할 | 컴포넌트 예시 |
|---|---|---|
| **무기 모듈** | 공격 방식 결정 | `UWeaponComponent` — 근접(회전 검), 원거리(투사체), 범위(오라) 등을 파생 클래스로 |
| **이동 모듈** | 이동 방식 결정 | `UMovementModifierComponent` — 대시, 텔레포트, 넉백 저항 등 |
| **생존 모듈** | 방어/회복 | `UHealthComponent`, `UShieldComponent`, `URegenComponent` |
| **패시브 모듈** | 능력치 증폭 | `UPassiveBuffComponent` — 이동속도 증가, 공격력 배율, 경험치 배율 등 |
| **적 행동 모듈** | 적 AI 패턴 | `UEnemyBehaviorComponent` — 추적형, 돌진형, 원거리형을 컴포넌트로 교체 |

**핵심 체험**: 같은 적 액터에 `추적형 행동 컴포넌트`를 붙이면 좀비처럼 따라오고, `돌진형`으로 바꾸면 일정 거리에서 돌격하고, `원거리형`으로 바꾸면 투사체를 발사한다. **코드를 바꾸는 게 아니라 컴포넌트를 교체하는 것만으로 행동이 달라진다**는 것을 직접 보여준다.

## GameMode가 빛나는 구간: 게임 모드 전환

### 왜 GameMode를 여기서 쓰는가

2주차에 배운 GameMode의 핵심 — **"게임의 규칙을 정의하는 객체"**. 하지만 이론만으로는 체감이 어려웠다. 이 프로젝트에서는 **같은 맵, 같은 캐릭터인데 GameMode만 바꾸면 게임이 완전히 달라지는 경험**을 만든다.

### 모드 예시

| GameMode | 규칙 | GameState가 관리하는 상태 |
|---|---|---|
| **GM_Survival** (기본) | 무한 웨이브, 최대 생존 시간이 기록 | 경과 시간, 현재 웨이브, 킬 수 |
| **GM_TimeAttack** | 제한 시간 내 최대 킬 수 경쟁 | 남은 시간, 킬 수, 점수 |
| **GM_BossRush** | 일반 몹 없이 보스만 연속 출현, 보스 클리어 횟수가 기록 | 현재 보스 번호, 클리어 수 |

**구현 포인트**:
- `AGameModeBase`를 상속한 3개 GameMode C++ 클래스
- 각 GameMode가 `DefaultPawnClass`, 스폰 규칙, 승/패 판정을 자체적으로 정의
- `AGameStateBase`를 상속해 모드별 상태(시간, 킬 수, 웨이브 등)를 관리
- **맵 로드 시 GameMode만 지정하면 전혀 다른 게임이 된다** → "아, GameMode가 이래서 필요한 거구나"

### 수업 연계

- 2주차 이론에서 배운 "GameMode = 룰, GameState = 상태" 공식을 실습으로 증명
- "하나의 GameManager에 모든 것을 넣는 God Object 안티패턴" vs "GameMode/GameState로 분리하는 언리얼 방식"을 직접 비교 체험

## 주차별 구현 로드맵

### 4주차 (4/27) — 기본 서바이벌 루프

**커리큘럼**: PlayerController, PlayerState, HUD

| 구현 항목 | 학습 주제와의 연결 |
|---|---|
| Enhanced Input으로 탑다운 WASD 이동 | PlayerController, Enhanced Input System |
| PlayerState에 HP, 경험치, 레벨 변수 선언 | PlayerState 역할 이해 — "플레이어 고유 상태의 보관소" |
| UMG로 HP 바 + 경험치 바 HUD 제작 | HUD/위젯 블루프린트 기초 |
| 기본 적(스피어) 스폰 + 플레이어 추적 이동 | AActor 스폰, Tick 기반 이동 (다음 주에 개선) |

**4주차 끝 결과물**: 캐릭터가 움직이고, 적이 따라오고, 맞으면 HP가 줄어드는 화면.

---

### 5주차 (5/4) — C++/BP 무기 시스템 + 적 클래스 분리

**커리큘럼**: 블루프린트 + 언리얼 C++ 문법

| 구현 항목 | 학습 주제와의 연결 |
|---|---|
| C++로 `UWeaponComponent` 기본 클래스 작성 | UPROPERTY, UFUNCTION 매크로 |
| `UWeaponComponent`를 상속한 3가지 무기 파생 | 상속 실습 (1주차 미진행분 보충) |
| — `UMeleeWeaponComp`: 캐릭터 주위 회전 공격 | |
| — `URangedWeaponComp`: 가장 가까운 적에게 투사체 | |
| — `UAOEWeaponComp`: 주변 범위 데미지 | |
| BP로 무기 파생 클래스 하나 추가 제작 | C++/BP 비교 체험 — "같은 결과, 다른 방식" |
| C++로 `AEnemyBase` 작성, BP로 Enemy 변형 제작 | 클래스 계층 설계 (UObject → AActor 복습) |

**5주차 끝 결과물**: 무기가 자동으로 적을 공격하고, 적이 2~3종으로 다양해진 서바이벌.

**모듈화 체험**: 캐릭터의 `UWeaponComponent`를 에디터에서 교체하면 공격 방식이 바뀌는 것을 직접 확인. "컴포넌트를 갈아끼우는 것만으로 행동이 변한다."

---

### 6주차 (5/11) — 델리게이트 통신 + 컴포넌트 완전 분리

**커리큘럼**: 델리게이트, 컴포넌트 설계, 오브젝트 간 통신

| 구현 항목 | 학습 주제와의 연결 |
|---|---|
| `UHealthComponent` 분리 — 플레이어와 적 모두에 부착 | 컴포넌트 기반 설계: "기능을 액터가 아닌 컴포넌트에" |
| `OnHealthChanged` 멀티캐스트 델리게이트 | 델리게이트 개념 + HUD 자동 갱신 |
| `OnEnemyDied` 델리게이트 → 경험치 지급 | 오브젝트 간 통신: 적 사망 이벤트를 누가 듣는가 |
| 경험치 트리거 볼륨 (보석 드롭 → 접촉 시 획득) | 트리거 볼륨 + Overlap 이벤트 |
| 레벨업 시 무기/패시브 선택 UI | 델리게이트로 레벨업 이벤트 → UI 팝업 |
| `UPassiveBuffComponent` 추가 (이속, 공격력 배율) | 패시브 능력도 컴포넌트로 — 붙이면 효과 발동, 떼면 해제 |

**6주차 끝 결과물**: 적을 죽이면 보석이 떨어지고, 레벨업하면 무기/패시브를 고르고, 점점 강해지는 서바이벌 루프 완성.

**모듈화 체험 심화**:
- `UHealthComponent`를 적에게도 똑같이 붙여서 "플레이어와 적이 같은 체력 시스템을 공유"하는 것을 확인
- `UPassiveBuffComponent`를 런타임에 `AddComponent`/`DestroyComponent`로 붙였다 떼는 실험 → "능력 획득 = 컴포넌트 부착, 능력 상실 = 컴포넌트 제거"

---

### 7주차 (5/18) — DataAsset으로 데이터 드리븐 전환

**커리큘럼**: 에셋 관리, DataAsset, Data-Driven 설계

| 구현 항목 | 학습 주제와의 연결 |
|---|---|
| `UWeaponDataAsset` — 무기 스탯(데미지, 쿨다운, 범위 등) | DataAsset 개념: "코드에 하드코딩하지 않는다" |
| `UEnemyDataAsset` — 적 스탯(HP, 이속, 공격력) | 같은 DataAsset 패턴으로 적도 데이터화 |
| `UWaveDataAsset` — 웨이브 구성(어떤 적이 몇 마리, 간격) | Data-Driven 설계의 정수: 코드 수정 없이 난이도 밸런싱 |
| SoftPtr로 무기 메시/이펙트 비동기 로딩 | TSoftObjectPtr, 비동기 로딩 체험 |
| 레벨업 선택 UI를 DataAsset 배열에서 동적 생성 | 에디터에서 DataAsset만 추가하면 게임에 새 무기가 등장 |

**7주차 끝 결과물**: 에디터에서 DataAsset을 하나 추가하면 게임 내에 새 무기가 자동으로 등장. 코드 한 줄 안 고치고 밸런스를 조정하는 체험.

**모듈화 완성**: 컴포넌트(행동) + DataAsset(데이터)가 분리되어 있으므로, **"어떻게 동작할지"는 컴포넌트가, "수치가 얼마인지"는 DataAsset이 결정**한다는 설계 원칙을 체감.

---

### 8주차 (5/25) — GameMode 전환 + 종합 마무리

**커리큘럼**: 코딩 컨벤션 + 프로젝트 종합

| 구현 항목 | 학습 주제와의 연결 |
|---|---|
| `GM_Survival` — 기본 무한 서바이벌 모드 정리 | GameMode 기본 구현 |
| `GM_TimeAttack` — 제한 시간 모드 추가 | 같은 게임, 다른 규칙 = GameMode 교체 |
| `GM_BossRush` — 보스 러시 모드 추가 (여유 시 도전) | GameMode의 확장성 체험 |
| GameState로 모드별 스코어/타이머 관리 | GameState의 역할 복습 |
| 네이밍 컨벤션 점검 + 폴더 구조 정리 | 코딩 컨벤션 실습 적용 |
| 메인 메뉴 → 모드 선택 → 플레이 → 결과 화면 흐름 | 전체 게임 루프 완성 |

**8주차 끝 결과물**: 메뉴에서 서바이벌/타임어택/보스러시 중 하나를 고르면 같은 맵에서 전혀 다른 규칙의 게임이 시작되는 완성된 프로젝트.

**GameMode 체험**: "3개의 게임을 만든 게 아니라, GameMode 3개를 만든 것이다" — 이것이 언리얼이 GameMode를 분리한 이유.

## 모듈화 설계의 전체 그림

```
Character (껍데기)
 ├── UWeaponComponent (교체 가능: 근접/원거리/범위)
 ├── UHealthComponent (공유 가능: 플레이어, 적 모두 사용)
 ├── UPassiveBuffComponent (런타임 부착/제거)
 └── UMovementModifierComponent (대시, 텔레포트 등)

Enemy (껍데기)
 ├── UHealthComponent (플레이어와 동일한 컴포넌트)
 ├── UEnemyBehaviorComponent (교체 가능: 추적/돌진/원거리)
 └── 스탯은 UEnemyDataAsset에서 로딩

GameMode (규칙만 교체)
 ├── GM_Survival: 무한 웨이브, 생존 시간 기록
 ├── GM_TimeAttack: 제한 시간, 킬 수 경쟁
 └── GM_BossRush: 보스만 등장, 클리어 수 기록

데이터 (코드 밖에서 관리)
 ├── UWeaponDataAsset: 무기 스탯
 ├── UEnemyDataAsset: 적 스탯
 └── UWaveDataAsset: 웨이브 구성
```

이 구조에서 수강생이 체감할 핵심 질문들:

1. **"왜 컴포넌트로 나눠야 하나요?"** → 같은 HealthComponent를 플레이어에도, 적에도, 나중에 파괴 가능한 오브젝트에도 붙일 수 있으니까
2. **"왜 GameMode를 따로 만들어야 하나요?"** → 같은 맵과 캐릭터로 3가지 다른 게임을 만들 수 있으니까
3. **"왜 DataAsset을 쓰나요?"** → 코드 한 줄 안 고치고 에디터에서 무기 밸런스를 바꿀 수 있으니까
4. **"왜 God Object는 안 되나요?"** → 이 모든 걸 하나의 클래스에 넣었다고 상상해보세요

## 비주얼 참고

아트 에셋 없이 프로토타입 가능한 방식:

- **플레이어**: 파란 캡슐 (기본 Character 메시)
- **적**: 빨간 스피어 (추적형), 주황 큐브 (돌진형), 보라 콘 (원거리형)
- **무기 이펙트**: 파티클 없이 디버그 도형(DrawDebugSphere, Line 등)으로 범위 표시
- **HUD**: UMG 기본 위젯 — 프로그레스 바(HP, EXP), 텍스트 블록(타이머, 킬 수)
- **맵**: 기본 평면 + BSP 또는 기본 메시로 울타리

복잡한 3D 모델이나 애니메이션 없이 **시스템 설계에만 집중**할 수 있는 구성이다.
