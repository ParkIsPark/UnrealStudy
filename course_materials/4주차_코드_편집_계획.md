## 1. 신규 생성 파일

### 1.1 `Source/Week3/Week3PossessSelector.h` (신규)

**목적:** 어떤 폰을 빙의(Possess) 대상으로 삼을지 레벨에서 선택하기 위한 빈 Actor. 실습 0에서 PlayerController가 `TActorIterator`로 검색해 사용한다.

**작성 내용:**
- `AActor` 상속, `WEEK3_API` 매크로
- `UPROPERTY(EditAnywhere, Category="Possess") APawn* CharacterToControl`
- 생성자/멤버 함수는 없음(데이터 컨테이너 역할)

**이유:** 코드 재컴파일 없이 디테일 패널에서 빙의 대상을 바꾸는 흐름을 학생들에게 보여주기 위함. PlayerController에 직접 하드코딩하지 않고 레벨 데이터로 분리해 "C++ vs 데이터" 경계를 체감시킨다.

### 1.3 `Source/Week3/Week3PlayerController.h` (신규)

**목적:** 실습 0의 핵심. `BeginPlay`에서 Selector를 찾아 `Possess` 호출.

**작성 내용:**
- `APlayerController` 상속
- `virtual void BeginPlay() override;` 만 선언
- 멤버 변수는 없음(Selector를 매번 검색)

**이유:** 학생이 "PlayerController가 폰을 능동적으로 차지한다"는 개념을 코드로 직접 보게 하는 것이 4주차 학습 목표. 따라서 GameMode에 의존하지 않고 PlayerController 자신이 검색→Possess 흐름을 갖는다.

### 1.4 `Source/Week3/Week3PlayerController.cpp` (신규)

**목적:** Selector 검색 + Possess 호출.

**작성 내용:**
- `#include "EngineUtils.h"` (TActorIterator)
- `#include "Week3PossessSelector.h"`
- `BeginPlay()` 본문:
  1. `Super::BeginPlay()`
  2. `for (TActorIterator<AWeek3PossessSelector> It(GetWorld()); It; ++It)`
  3. 첫 Selector의 `CharacterToControl`이 유효하면 `Possess(Selector->CharacterToControl)`
  4. 첫 매치 후 `break`

**이유:** PPT에서 강조한 "PlayerController는 Pawn을 갈아탈 수 있다"는 핵심 메시지를 Possess 한 줄로 보여준다. 검색 비용은 BeginPlay에서 한 번뿐이고, 학습용으로 충분히 단순하다.

### 1.5 (옵션·권장) `Source/Week3/Week3PlayerState.h` (신규)

**목적:** 권장 경로에서 PlayerState를 C++로 선언해 캐릭터 코드의 캐스팅을 단순·견고하게 만든다.

**작성 내용:**
- `APlayerState` 상속
- `UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Stats") float MaxHP = 100.f;`
- `UPROPERTY(BlueprintReadOnly, Category="Stats") float CurrentHP = 100.f;`
- `UFUNCTION(BlueprintCallable, Category="Stats") void TakeDamage(float DamageAmount);`
- `UFUNCTION(BlueprintCallable, BlueprintPure, Category="Stats") bool IsAlive() const;`
- `virtual void BeginPlay() override;` (CurrentHP=MaxHP 초기화)

**이유:**
1. `Cast<AWeek3PlayerState>(GetPlayerState())` 한 줄로 PlayerState 함수 호출 가능 → 4주차 핵심인 "Pawn↔PlayerState 데이터 흐름"을 깨끗한 C++ 코드로 시연 가능.
2. 실습 계획서 Step 5의 `ProcessEvent`+`UFunction*` 우회는 동작은 하지만 학생이 따라하기 어렵고 BP 클래스명에 결합된다. 강의에서는 위험성만 언급하고 권장 경로로 가는 편이 교육적.
3. BP_PlayerState를 이 클래스의 자식으로 두면 Editor 에서 값 조정과 BP 함수 추가도 그대로 가능 → BP-only 경로의 장점을 잃지 않는다.

**BP-only 경로를 따를 때:** 이 파일은 만들지 않는다.

### 1.6 (옵션·권장) `Source/Week3/Week3PlayerState.cpp` (신규)

**작성 내용:**
- 생성자: `MaxHP=100.f`, `CurrentHP=100.f`
- `BeginPlay`: `Super::BeginPlay(); CurrentHP = MaxHP;`
- `TakeDamage`: `CurrentHP = FMath::Clamp(CurrentHP - DamageAmount, 0.f, MaxHP);`

**이유:** 헤더 짝 + 단순 산술 로직. Clamp 사용으로 음수/오버플로 방지를 코드로 시연.

```cpp
// 권장 경로 (C++ PlayerState 사용)
void AWeek3Character::DebugDamage(const FInputActionValue&)

    }

// BP-only 경로일 때 (실습 계획서 그대로)
void AWeek3Character::DebugDamage(const FInputActionValue&)
{
    APlayerState* PS = GetPlayerState();
    if (!PS) return;
    UFunction* Fn = PS->FindFunction(FName("TakeDamage"));
    if (!Fn) return;
    struct { float DamageAmount; } Params{ 10.f };
    PS->ProcessEvent(Fn, &Params);
}
```

#### 2.3.4 생성자 수정 (선택)

`GetCharacterMovement()->MaxWalkSpeed = 500.f;`은 그대로 두되, 강의에서 "Walk/Sprint 속도가 두 변수로 분리되었다"는 점을 강조하기 위해:

```cpp
// 기존: GetCharacterMovement()->MaxWalkSpeed = 500.f;
GetCharacterMovement()->MaxWalkSpeed = WalkSpeed;
```

로 바꾸거나, 그냥 `WalkSpeed = 500.f` 기본값과 일치시켜 두 줄 모두 살려도 무방. **권장:** `WalkSpeed`를 단일 진실원으로 사용해 한 줄로 정리.

**이유:** 같은 값(500)이 두 곳에 있으면 디테일 패널에서 학생이 WalkSpeed를 바꿨는데 게임이 안 변하는 혼란을 막는다.

- 기존 `IMC_Default`의 위치는 `Content/ThirdPerson/Input/`이므로 신규 IA들도 같은 트리(`Actions/`)에 두어 컨벤션 유지.
- `WEEK3_API` 매크로는 모듈 외부 노출이 필요한 클래스에만 붙인다(여기서는 모두 동일 모듈 내이므로 일관성 위해 붙여도 무방).
- `*.generated.h`는 항상 마지막 include.
- BP-only 경로의 `ProcessEvent`는 함수명 문자열이 컴파일 타임에 검증되지 않는다는 점을 강의 중 명시.
- `Default Pawn Class = None`을 잊으면 자동 스폰 폰이 카메라를 빼앗아 Possess 흐름이 보이지 않는다.
- `BP_Gamemode`에 `Player Controller Class`를 BP가 아닌 C++ 클래스로 지정하면 BP에서 추가한 위젯 생성 노드가 실행되지 않는다 — 반드시 BP 자식을 지정.
- 빌드 모듈에 `UMG` 추가 후 첫 빌드 시간이 늘어나는 점을 학생에게 미리 안내.
